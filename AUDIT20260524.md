# Accessibility Project Manager ‚Äî Audit Report

Audit against four stated requirements:

1. Track generation and distribution of all material types (braille, large print, eBraille, EPUB3/DAISY, tactile graphics, 3-D print)
2. All steps auditable
3. Clean, understandable organization, database structure, and file storage
4. Find any step of any project from any available piece of metadata

## Requirement 1: Track All Material Types

### What works

Every material type has a dedicated database table, workflow stages, UI page, and the ability to link files and metadata. The job type coverage is complete:

- Braille (literary, math/Nemeth, science, music) with embosser assignment
- Large Print
- eBraille
- EPUB3/DAISY (first-class, not lumped with eBraille)
- Tactile Graphics (thermoform/SWELL, hand-tooled, embossed figures)
- 3-D Print Jobs

### Critical gap: Print jobs have no workflow step tracking

Every other job type has discrete, auditable workflow stages stored as boolean columns with `complete_step`/`revert_step` functions that log events. Print jobs have none of this.

The `workflow_step` table is seeded with print stages (Designed, Sliced, Printed, Inspected, Delivered) and the `print_job` table could support step columns, but neither the schema nor the UI implements them. A print job is a single log entry ‚Äî you can record it happened, but you cannot track it moving through fabrication stages or attach files at each stage.

**Fix:** Add step columns to `print_job` (or a linked `print_job_step` table), add step completion UI to the print jobs detail view, and call `log_event` at each transition.

### Critical gap: Tactile graphics jobs have no file attachment UI

The `job_file_link` table supports any job type, and `Q.list_files_for_job` and `Q.link_file_to_job` work for tactile jobs. But `tactile_graphics.py` has no `_ingest_dialog` function and no "Attach File" button anywhere in the detail view.

This means you cannot attach the source image, the SWELL file, or the final tactile graphic to the job record. There is no file provenance trail for tactile graphics jobs.

**Fix:** Add the same `_ingest_dialog` pattern used in `braille_jobs.py` to `tactile_graphics.py`.

### Minor gap: No distribution/delivery tracking

Once a job reaches "Delivered" status, there is no record of *how* it was delivered, *to whom* it was physically given, or *when* it was received by the student/patron. The delivery step is a boolean checkbox, not a delivery record.

**Fix:** Add optional delivery fields to the job tables: `delivery_date`, `delivery_method` (e.g., email, physical, learning management system), `delivery_recipient`, and `delivery_notes`. Log a DELIVERY event when the step is completed.

## Requirement 2: All Steps Auditable

This is where the project has its most serious gaps. The audit infrastructure exists and is well-designed ‚Äî the `metadata_event` table is purpose-built for PREMIS-style event logging ‚Äî but it is inconsistently used.

### What works

The following operations correctly log events to `metadata_event`:

- Job creation (braille, LP/eBraille, tactile, print) ‚Äî logged as `CREATE`
- Workflow step completion ‚Äî logged as `STEP_COMPLETE`
- Workflow step reversion ‚Äî logged as `STEP_REVERT`
- File ingestion ‚Äî logged as `INGEST` (when done through the dedicated ingestion page)
- File-job linking from braille and LP job detail views ‚Äî logged as `INGEST`
- Manual note events ‚Äî logged as `NOTE`
- QA tool runs ‚Äî logged to `qa_run` table
- Pipeline step runs ‚Äî logged to `pipeline_step_run` table

### Critical gap: Edit operations generate no audit record

All four update functions ‚Äî `update_braille_job`, `update_lp_job`, `update_tactile_job`, `update_print_job` ‚Äî execute a SQL UPDATE and return nothing. No `log_event` call exists anywhere after them.

If a requester name is corrected, a due date changed, a priority escalated to urgent, or a title fixed, there is no record that any of this happened, what it was before, or who changed it. The `updated_at` timestamp changes, but that tells you nothing about what the change was.

**Fix:** After each update call, log a `FIELD_UPDATE` event with the changed fields serialized into `extra_metadata`. Example:
```python
def update_braille_job(row_id: int, **fields: Any) -> None:
    old = get_braille_job(row_id)
    sql, vals = _build_update_sql("braille_job", fields, allowed)
    with get_conn() as conn:
        conn.execute(sql, vals + [row_id])
    log_event("braille", row_id, "FIELD_UPDATE", "SUCCESS",
              extra_metadata={"changed": list(fields.keys()),
                              "previous": {k: old.get(k) for k in fields if old}})
```

### Critical gap: Delete operations generate no audit record

`delete_braille_job`, `delete_lp_job`, `delete_tactile_job`, and `delete_print_job` issue a bare `DELETE` with no prior event log entry. A job ‚Äî including all its metadata, file links, and event history ‚Äî can be permanently removed with zero trace in the audit log.

**Fix:** Log a `DELETE` event *before* the DELETE executes, capturing the job title and key fields in `extra_metadata`. Since the event log is scoped to a job that will no longer exist, consider also writing a system-level "tombstone" record to a separate `deleted_jobs` log table.

### Critical gap: Metadata saves generate no audit record in the database

The `set_job_metadata` function (called from the metadata editor and job detail dialogs) performs an upsert with no event logging. The `MetadataAuditService` in `metadata_editor.py` does capture before/after snapshots, but it stores them in a Python list (`_events: list[MetadataAuditEvent] = []`). This data lives in process memory and is lost on every restart.

The `metadata_event` table is the right place for this. It is never called when metadata is saved from any of the job detail dialogs (`braille_jobs.py`, `lp_ebraille.py`, `print_jobs.py`, `tactile_graphics.py`).

**Fix:** After the metadata save block in each dialog's `_save_all` function, call:
```python
Q.log_event(job_type, job_id, "METADATA_UPDATE", "SUCCESS",
            agent="user",
            detail=f"Updated {len(saved_keys)} metadata fields",
            extra_metadata={"keys": saved_keys})
```

### Gap: File attachment from job detail views bypasses project artifact storage

The `_ingest_dialog` in `braille_jobs.py` and `lp_ebraille.py` calls `Q.ingest_file` without passing `project_title`, `student_initials`, `school_name`, `grade_level`, or `subject`. This means files attached directly from within a job's detail view are stored as `job_files/<uuid>.<ext>` ‚Äî the legacy UUID-named path ‚Äî rather than in the structured `artifacts/<Project Title>/` hierarchy.

Only the standalone File Ingestion page uses the artifact path convention. This creates two classes of files in the database: some with human-readable structured paths and some with opaque UUIDs, depending on which UI was used.

**Fix:** Pass the job's existing metadata (fetched from `job_metadata`) as the project context when calling `Q.ingest_file` from job detail dialogs, or add project context fields to the ingest dialog.

### Gap: QA runs are not linked to jobs

When a QA tool is run from the QA Tooling page, `job_type=None` and `job_id=None` are passed to `Q.log_qa_run`. The run is recorded in `qa_run` but has no connection to any specific job. If you run EPUBCheck on an EPUB file that belongs to a specific EPUB3/DAISY job, that validation result does not appear in the job's event log.

**Fix:** Add a "link to job" option in the QA run dialog (same pattern as the file ingestion dialog), and log a corresponding `QA_RUN` event to `metadata_event` for the linked job.

## Requirement 3: Clean, Understandable Organization

### Critical gap: Two separate database files

The most significant organizational problem in the project is that there are two different SQLite database files being created and written to simultaneously, using two entirely separate connection systems.

**The main database** (`accessibility_manager.db`) is managed by `db/schema.py` and `db/queries.py`. This is where all jobs, files, metadata, events, inventory, and audit records live.

**A shadow database** (`accessibility_mgr.db`) is created by `services/sqlite_store.py`. The following services write exclusively to this second database, completely isolated from the main one:

- `PersistentAnalyticsService` ‚Üí `analytics_metric` table in `accessibility_mgr.db`
- `PersistentProvenanceRegistry` ‚Üí `provenance_event` table in `accessibility_mgr.db`
- `PersistentQAService` ‚Üí `qa_report` and `pipeline_run` tables in `accessibility_mgr.db`
- `DistributedQueueService` ‚Üí `distributed_workflow_queue` table in `accessibility_mgr.db`
- `MigrationManager` ‚Üí `schema_version` table in `accessibility_mgr.db`

None of this data is visible in the main database, none of it is included in the automated backups (which back up `accessibility_manager.db`), and none of it can be queried alongside the main operational records.

Note also that `services/persistent_qa.py` defines a `pipeline_run` table in `accessibility_mgr.db` with different columns from the `pipeline_run` table in `accessibility_manager.db`. Two tables with the same name, different schemas, in different files.

**Fix:** Delete `services/sqlite_store.py` and all services that depend on it. Either discard these services (most are unused in the actual UI) or rewrite them to use `db/queries.py` and the main database connection.

### Critical gap: Dead legacy code creates false complexity

The following files are legacy from a previous SQLAlchemy-based architecture. They are not used by any active UI page and create confusion about which system to use:

- `db/database.py` ‚Äî creates a second SQLAlchemy engine pointing to `accessibility_manager.db`
- `models/assets.py` ‚Äî SQLAlchemy ORM models (Project, Asset, AssetMetadata, WorkflowEvent)
- `models/inventory.py` ‚Äî SQLAlchemy ORM models (Category, InventoryItem, InventoryTransaction)
- `services/assets_service.py` ‚Äî wraps the SQLAlchemy models
- `services/inventory_service.py` ‚Äî wraps the SQLAlchemy models
- `services/file_ingestion_service.py` ‚Äî uses AssetService
- `services/package_service.py` ‚Äî uses AssetService
- `services/search_service.py` ‚Äî uses SQLAlchemy Session queries
- `services/workflow_engine.py` ‚Äî uses AssetService and WorkflowEvent
- `ui/asset_detail.py` ‚Äî uses AssetService
- `ui/assets.py` ‚Äî uses AssetService
- `ui/categories.py` ‚Äî uses SQLAlchemy Session directly
- `ui/inventory.py` ‚Äî uses InventoryService
- `ui/workflow_braille.py` ‚Äî uses WorkflowEngine
- `ui/workflow_printing.py` ‚Äî uses WorkflowEngine

These are registered as imports in some modules but none appear in the `PAGE_DEFINITIONS` in `app.py` (except through the `__init__.py` legacy import aliases). They make the codebase approximately 40% larger than it needs to be while providing no running functionality.

**Fix:** Delete all of the above. If sqlalchemy is not in active use, remove it from `requirements.txt` as well.

### Gap: `file_use` vocabulary is inconsistent

The `material_category` table is seeded with file use values: `MASTER`, `DERIVATIVE`, `INTERMEDIATE`, `SOURCE`, `REFERENCE`.

The `_FILE_USES` list in `braille_jobs.py` and `lp_ebraille.py` contains: `ORIGINAL`, `DERIVATIVE`, `INTERMEDIATE`, `SOURCE`, `REFERENCE`.

The ingestion page defaults to `ORIGINAL`.

`MASTER` and `ORIGINAL` are different values. Records in the database will have inconsistent `file_use` values depending on which UI path was used to ingest the file.

**Fix:** Pick one ‚Äî `MASTER` aligns with METS/PREMIS conventions; `ORIGINAL` is more intuitive. Update both the seed data in `schema.py` and the `_FILE_USES` lists in all UI files to match.

### Gap: No student or requester entity

The `requester` field on every job table is a plain text string. There is no `student` or `requester` table with a primary key. This means:

- "Smith, John" and "John Smith" and "J. Smith" are three different requester values with no way to know they are the same person.
- You cannot pull all jobs for a given student without exact string matching.
- There is no place to store student-level information (school, grade, IEP status, preferred formats) that automatically populates new jobs.

**Fix:** Add a `student` table with name, school, grade, preferred formats, and notes. Add a `student_id` foreign key to all job tables. The current requester/school/grade fields on job records are useful for per-job context but should supplement, not replace, a student record.

## Requirement 4: Find Any Step From Any Metadata

### What works

The Search page queries job titles, requester names, filenames, format names, encodings, and metadata key/value pairs. The Lineage Viewer generates a provenance graph from file-to-job links and displays the unified event timeline. The `job_metadata` key/value structure is flexible enough to store any piece of identifying information.

### Critical gap: Search loads all records into Python memory

The `search_page` function in `ui/search.py` calls:

```python
Q.list_braille_jobs()      # all braille jobs
Q.list_lp_jobs()           # all LP/eBraille/EPUB3 jobs
Q.list_tactile_jobs()      # all tactile jobs
Q.list_print_jobs()        # all print jobs
Q.list_file_objects()      # all files
```

...and then iterates every record in Python using `_match(text, query)` (a case-insensitive `in` check). For a large dataset this will be slow and memory-intensive. More importantly, the metadata search queries every single job's metadata separately in a loop:

```python
for jtype, jid in all_job_ids:
    for k, v in Q.list_job_metadata(jtype, jid).items():
```

This is O(n) database round trips where n is the number of jobs. With 500 jobs this is 500+ sequential queries just for the metadata portion.

**Fix:** Replace in-memory filtering with parameterized SQL:

```sql
-- Job search
SELECT id, title, 'braille' AS job_type FROM braille_job
WHERE title LIKE ? OR requester LIKE ? OR notes LIKE ?
UNION ALL
SELECT id, title, 'lp_ebraille' FROM lp_ebraille_job
WHERE title LIKE ? OR requester LIKE ? OR notes LIKE ?

-- Metadata search
SELECT jm.job_type, jm.job_id, jm.meta_key, jm.meta_value
FROM job_metadata jm
WHERE jm.meta_key LIKE ? OR jm.meta_value LIKE ?
```

A single parameterized query across all tables with `LIKE '%term%'` is orders of magnitude faster than Python-side filtering of fully-loaded result sets.

### Gap: Search cannot reach event log content

The event log in `metadata_event` contains free-text `detail` fields and JSON `extra_metadata`. If an operator adds a note saying "Student confirmed receipt on 2026-03-15", that text is not searchable from the Search page. Neither are agent names, event types, or linked file names from events.

**Fix:** Add `metadata_event` to the search query. The `detail` and `agent` columns are the most useful targets.

### Gap: Search cannot find files by checksum or stored path

If you have a SHA-256 hash of a file and want to find which job it belongs to, that is not possible from the Search page. The `checksum_sha256` and `stored_path` columns exist in `file_object` but are not included in search.

**Fix:** Include `checksum_sha256` and `stored_path` in the file search clause.

### Gap: No cross-job reporting by student or school

There is no page or query that answers "show me everything produced for Legacy Jr. High this year" or "show me all active jobs for students in Grade 7." The search can find individual jobs with that text in the title or requester field, but there is no aggregate view.

**Fix:** Add a reporting/filter page with multi-field faceted search: school name, grade level, material type, status (any step complete, fully delivered, in progress), date range. This is the most operationally useful view for production planning.

### Gap: The structural map is built but never used

The `structural_map_node` table and its query functions (`add_struct_node`, `list_struct_nodes`, `delete_struct_node`) exist in the schema and data layer but there is no UI that creates or displays structural map nodes. For multi-volume braille sets or DAISY books with complex chapter structures, this table was designed to represent the document hierarchy. Currently it is empty and unreachable.

This is not urgent for simple single-file jobs, but for a 10-volume math textbook in braille, there is currently no way to represent the relationship between the volumes, their chapters, and the files at each level.

## Additional Issues

### Backup status is not visible in the UI

`BackupService.status()` returns backup count, last backup path, size, directory, and scheduler state. This information is never surfaced in the Admin panel or anywhere else in the UI. There is also no button to trigger a manual backup from the UI. If the scheduler silently fails (e.g., because the database did not exist yet at the 30-second mark), there is no indication of this.

**Fix:** Add a "Backups" section to the Admin page showing the last N backups from `list_backup_log()`, last backup timestamp, file size, and a "Run Backup Now" button.

### The Admin page has no metadata key backfill trigger that is easily discoverable

The typo backfill tool exists (`Q.backfill_metadata_keys`) and there is a button for it in the Metadata Options tab of Admin. However, there is no explanation of what it does or when to use it, and no preview of what will change before confirming. A misconfigured run could silently merge keys that should not be merged.

**Fix:** Add a dry-run preview step that shows the proposed key mappings before the user confirms, and display a detailed result summary after the run.

### `workflow_monitor.py` starts a background worker thread on import

`ui/workflow_monitor.py` calls `_runtime.start(_mock_handler)` at module level. This means importing or loading this file starts a daemon thread. While it is a daemon thread and will not block shutdown, it runs `time.sleep(0.2)` in a loop processing mock jobs. This file is not in `PAGE_DEFINITIONS` in `app.py` so it does not currently load, but if it were added, it would start a background thread that processes and completes the seeded queue items immediately.
